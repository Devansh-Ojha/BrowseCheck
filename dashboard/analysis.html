Root Cause of False Positives

The naive system prompt and the optimized version both flag benign inputs because the _MARKERS regex is too broad. Keywords like "authenticate", "password", and "credentials" are flagged even when they appear in legitimate technical documentation, not adversarial instructions.

Regex markers causing FP

Pattern Intention False Positive Example Severity
authenticate Catch "reauthenticate now" "Authenticate with K8s cluster" HIGH
password Catch "change your password to X" "Use username/password auth" HIGH
credentials Catch "send credentials to X" "Store credentials safely" HIGH
ignore|disregard Catch "ignore previous" None in examples LOW

Recommended fix

Change the `_MARKERS` regex to require instruction context

Instead of matching single keywords, match multi-word instruction patterns:

# Current (too broad):

r"ignore|disregard|password|credentials|authenticate|..."

# Recommended (requires instruction context):

r"(?:ignore|disregard|instead|you must|bypass).*(?:instructions|task|prompt|system|previous)"

Result: Still catches "ignore previous instructions" but NOT "Authenticate with K8s" or "store credentials safely".
